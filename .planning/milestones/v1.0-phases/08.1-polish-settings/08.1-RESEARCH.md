## Summary

This phase is a UI polish pass that touches all five games and the welcome screen. There are four distinct workstreams: (1) adding drag-to-select to Woord Soek by extending existing touch handlers, (2) applying MS Solitaire-style green felt backgrounds and card backs to the three card games plus a parchment theme for Woord Soek, (3) ensuring the font size control on the welcome screen actually propagates correctly to all game text, and (4) standardizing settings modals across all five games.

The codebase is well-structured for these changes. The existing `shared.css` CSS variable system supports theme overrides via CSS classes on game containers. The `Settings.js` module already has `applyFontSize()` with small/medium/large mapping. The welcome screen already has Klein/Medium/Groot buttons wired up in the settings panel. The existing settings modals in Solitaire, Spider, and FreeCell use a consistent `showOverlay()`/`hideOverlay()` pattern with `.win-overlay` containers. Sudoku and Woord Soek need new settings modals built to match.

The key risk is font size propagation -- many game elements use `px` or `clamp()` with `rem` base, so switching the root font-size should scale them, but card dimensions and grid cells must remain fixed. A careful audit of which CSS properties use `rem`/`em` vs `px` is needed.

**Primary recommendation:** Structure work as three plans: (1) CSS themes + card backs (visual), (2) drag-to-select for Woord Soek (interaction), (3) font size propagation + settings modal standardization (settings).

### Pattern 2: Drag-to-Select via touchstart/touchmove/touchend
**What:** Extend the existing Woord Soek touch handling to support drag completion.
**When to use:** For the drag-to-select feature.
**How it works:**
1. `touchstart` on a cell sets `_firstCell` and enters `dragging` state
2. `touchmove` shows preview line (existing `showPreviewTo` function)
3. `touchend` reads the final cell under the finger, calls `checkSelection`, and if a word is found, highlights it
4. Both tap-tap AND drag work simultaneously -- the state machine must handle both

**Key insight:** The existing `onCellTap` uses `click` events, while drag uses `touchstart`/`touchmove`/`touchend`. These are separate event paths. The touch handlers need a `_isDragging` flag that's set to `true` on `touchmove` (after some threshold), and `touchend` only completes a drag if `_isDragging` is true. If the user doesn't move far enough, it falls through to the `click` handler for tap-tap.

```javascript
// Skeleton for drag-to-select integration
var _isDragging = false;
var _dragStartCell = null;

function onGridTouchStart(e) {
  var touch = e.touches[0];
  var target = document.elementFromPoint(touch.clientX, touch.clientY);
  if (!target || !target.classList.contains('ws-cell')) return;

  _dragStartCell = {
    row: parseInt(target.getAttribute('data-row'), 10),
    col: parseInt(target.getAttribute('data-col'), 10)
  };
  _isDragging = false;
}

function onGridTouchMove(e) {
  if (!_dragStartCell) return;
  var touch = e.touches[0];
  var target = document.elementFromPoint(touch.clientX, touch.clientY);
  if (!target || !target.classList.contains('ws-cell')) return;

  var r = parseInt(target.getAttribute('data-row'), 10);
  var c = parseInt(target.getAttribute('data-col'), 10);

  // If moved 2+ cells away, consider it a drag
  var dist = Math.max(Math.abs(r - _dragStartCell.row), Math.abs(c - _dragStartCell.col));
  if (dist >= 2) _isDragging = true;

  if (_isDragging) {
    // Set first cell if not set in tap-tap state
    if (_selectionState !== 'first_selected') {
      _firstCell = _dragStartCell;
      _selectionState = 'first_selected';
      addClassToCell(_dragStartCell.row, _dragStartCell.col, 'selected');
    }
    showPreviewTo(touch.clientX, touch.clientY);
  }
}

function onGridTouchEnd(e) {
  if (_isDragging && _firstCell) {
    // Find the cell under last touch position
    // Use the last preview position to determine end cell
    // Call checkSelection and handle result
    // ...
    clearSelection();
  }
  _isDragging = false;
  _dragStartCell = null;
  clearPreview();
}
```

### Anti-Patterns to Avoid
- **Inline background colours in game HTML:** Use CSS classes/variables. Theme changes should be one CSS class toggle, not scattered inline styles.
- **Preventing click events during drag:** Don't use `e.preventDefault()` on touchstart -- it breaks the click fallback for tap-tap users. Instead, use a flag to skip the click handler when a drag was detected.
- **Scaling card dimensions with rem:** Card widths/heights use `px`, `vw`, or `clamp()` with viewport units. Do NOT convert these to `rem` or they'll scale with font size and break layouts.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Touch event coordination | Custom gesture recognizer | Simple `_isDragging` flag + distance threshold | Overcomplicated gesture libs add bundle size; the existing pattern just needs extension |
| CSS noise/texture | Image files | SVG data URIs in CSS `background-image` | Already used in `body::before` for noise texture; keep pattern consistent |
| Settings persistence | Custom storage layer | Existing `Settings.get()/set()` | Already handles JSON serialization, prefixed keys, error handling |

**Key insight:** The codebase already has all the infrastructure. This phase is wiring + CSS, not new architecture.

## Common Pitfalls

### Pitfall 1: Touch Event Ghost Clicks
**What goes wrong:** On mobile browsers, a `touchend` event is followed by a synthetic `click` event ~300ms later. If drag-to-select completes on `touchend`, the subsequent `click` may trigger an unwanted tap-tap action.
**Why it happens:** Browser compatibility layer for sites that only handle click events.
**How to avoid:** Set a `_justCompletedDrag` flag on `touchend` that's cleared after a short timeout (~400ms). In the `click` handler, check this flag and skip if true.
**Warning signs:** Double selection or unexpected cell highlights after a successful drag.

### Pitfall 2: elementFromPoint Returns Wrong Target During Fast Drag
**What goes wrong:** `document.elementFromPoint()` during `touchmove` may return a child element or a non-cell element.
**Why it happens:** Touch coordinates don't always align with cell boundaries; fingers are imprecise.
**How to avoid:** Use `.closest('.ws-cell')` on the result, or check `classList.contains('ws-cell')` and walk up if needed. The existing code already checks for `.ws-cell` class.
**Warning signs:** Preview line flickering or disappearing during drag.

### Pitfall 3: Font Size Breaking Card Layouts
**What goes wrong:** Changing root `font-size` (16/18/21px) causes card stacking, modal sizing, or game layouts to shift if they use `rem` units.
**Why it happens:** `rem` is relative to root font-size. Anything sized in `rem` will scale.
**How to avoid:** Audit all game container sizes. Card dimensions, grid cells, and layout containers should use `px`, `vw`, or `clamp()` with viewport units. Only text labels, headers, word list items, and number pad text should use `rem`/`em`. The current codebase mostly does this correctly already -- card CSS uses `clamp(0.7rem, 1.2vw, 1.1rem)` for corner text, which scales slightly with font size. This is acceptable per CONTEXT.md: "card corner text is Claude's discretion."
**Warning signs:** Cards overlapping or game area overflowing when font size is set to "Groot."

### Pitfall 4: Parchment Theme Breaking Text Contrast
**What goes wrong:** Switching Woord Soek to a light parchment background with dark text, but some elements (found word highlights, selected cells) still use the warm-dark colour scheme and become invisible.
**Why it happens:** Not all CSS variables are overridden in the parchment theme.
**How to avoid:** Override ALL colour variables that affect text and interactive elements within the parchment theme class. Test all states: default, selected, preview, found, hint-flash.
**Warning signs:** Text disappearing or low-contrast elements on the parchment background.

### Pitfall 5: Settings Modal Z-Index Conflicts
**What goes wrong:** Settings modal appears behind win overlay or difficulty modal.
**Why it happens:** Multiple overlays use `.win-overlay` with `z-index: 500`. When two are active simultaneously, order depends on DOM position.
**How to avoid:** Only one overlay should be active at a time. The existing pattern already works this way -- settings close before win opens, etc. Maintain this discipline.
**Warning signs:** Clicking through an overlay to another overlay underneath.

### Pitfall 6: Woord Soek Gear Icon Conflicts with Existing Difficulty Icon
**What goes wrong:** Woord Soek header already has a gear icon (&#9881;) for the difficulty button. Adding another gear for settings creates confusion.
**Why it happens:** The difficulty button was assigned the gear icon before a standardized settings button was planned.
**How to avoid:** Change the Woord Soek difficulty button icon to something else (e.g., the bar chart icon or a difficulty-specific icon), then use gear exclusively for settings. Or use a different icon for the settings modal.
**Warning signs:** Two identical gear icons in the Woord Soek header bar.

### Drag-to-Select Touch End Handler
```javascript
function onGridTouchEnd(e) {
  if (!_isDragging || !_firstCell) {
    // Not a drag -- let click handler deal with tap-tap
    _isDragging = false;
    _dragStartCell = null;
    clearPreview();
    return;
  }

  // Use last known touch position
  var lastR = _lastPreviewRow;
  var lastC = _lastPreviewCol;

  if (lastR !== null && lastC !== null) {
    var dir = WoordSoekEngine.getDirection(_firstCell.row, _firstCell.col, lastR, lastC);
    if (dir) {
      var result = WoordSoekEngine.checkSelection(
        _firstCell.row, _firstCell.col, lastR, lastC, _puzzle.placedWords
      );
      if (result.found && !_foundWords.has(result.word)) {
        // Word found via drag!
        _foundWords.add(result.word);
        _foundCount++;
        var pw = _puzzle.placedWords[result.index];
        highlightFoundWord(result.word, pw.cells);
        markWordInList(result.word);
        el('ws-counter').textContent = _foundCount + ' / ' + _puzzle.placedWords.length;
        Audio.play('word_found');
        if (_foundCount === _puzzle.placedWords.length) {
          onPuzzleComplete();
        }
      }
    }
  }

  // Reset
  clearSelection();
  _isDragging = false;
  _dragStartCell = null;
  _justCompletedDrag = true;
  clearPreview();
  setTimeout(function() { _justCompletedDrag = false; }, 400);
}
```

### Font Size Audit Checklist
```
Items that SHOULD scale with font size (use rem/em):
- Game headers (h1, h2 text)
- Button labels ("Nuwe Spel", "Ontdoen")
- Word list items (.ws-word-item)
- Sudoku number pad digits (.sdk-num-btn font-size)
- Sudoku pencil marks (.sdk-note-digit font-size)
- Timer display (.timer)
- Badge text (.badge)
- Modal text content
- Woord Soek counter

Items that MUST NOT scale (use px/vw/clamp-with-vw):
- Card width/height (aspect-ratio + max-width in px)
- Woord Soek grid cell size (aspect-ratio, grid auto-fit)
- Sudoku grid cell size
- Game container padding/layout
- Card corner text (controlled, slight scaling OK)
- Column widths (sol-column, spd-column, fc-column max-width)
- Tap target minimum (--tap-min: 56px stays px)
```
